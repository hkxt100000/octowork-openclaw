## 🌐 后端服务概览

**技术栈**: Node.js + Express + SQLite
**核心服务**: API接口、WebSocket实时通信、消息处理、任务监控
**运行端口**: 1314 (开发版)
**API基础路径**: `http://localhost:1314/api`

**核心功能**:
- Bot管理: 获取Bot列表、状态监控
- 消息处理: 发送/接收消息、消息协调、总结服务
- 群组管理: 群聊创建、成员管理、文件树获取
- 任务监控: 定时任务执行、状态检查
- 文件操作: 文件上传、下载、管理
- WebSocket: 实时消息推送、连接管理

## 📁 模块详解

### 1. **主服务文件** (`server.js`)
**核心文件**: 所有API路由、中间件、服务初始化的入口。

**主要功能**:
- Express应用配置 (CORS、JSON解析、静态文件服务)
- RESTful API路由定义 (Bot、消息、群组、任务、文件等)
- 数据库连接初始化
- WebSocket服务器集成
- 定时任务启动
- 错误处理和日志记录

**关键API端点**:
- `GET /api/bots` - 获取所有Bot列表 (从AI Directory)
- `GET /api/bots/:id/messages` - 获取Bot消息历史
- `POST /api/bots/:id/messages` - 向Bot发送消息
- `GET /api/groups` - 获取群组列表
- `GET /api/groups/:groupId/file-tree` - 获取群组文件树
- `POST /api/summarize/message` - 总结消息内容
- `GET /api/health` - 健康检查

**端口配置**: 监听1314端口 (开发版标准端口)

---

### 2. **API模块** (`api/`)
**职责**: 封装对外部服务的API调用，主要是OpenClaw网关。

```
api/
└── 📄 openclaw.js              # OpenClaw网关API调用
    ├── callOpenClaw()          # 通用OpenClaw调用函数
    ├── sendMessageToBot()      # 发送消息到指定Bot
    ├── getBotStatus()          # 获取Bot状态
    └── executeCommand()        # 执行OpenClaw命令
```

**依赖关系**: 被 `server.js` 中的路由处理器调用。

---

### 3. **业务服务层** (`services/`)
**职责**: 核心业务逻辑实现，独立于HTTP层。

```
services/
├── 📄 task-monitor.js              # 任务监控服务
│   ├── monitorTasks()              # 监控所有任务状态
│   ├── checkTaskHealth()           # 检查任务健康度
│   └── restartFailedTasks()        # 重启失败的任务
│
├── 📄 message-reconciliation-enhanced.js  # 消息协调服务 (增强版)
│   ├── reconcileMessages()          # 协调消息状态
│   ├── findUnpushedBotMessages()    # 查找未推送的Bot消息
│   ├── markMessageAsPushed()        # 标记消息为已推送
│   └── processPendingMessages()     # 处理待处理消息
│
├── 📄 messageSummaryService.js      # 消息总结服务
│   ├── summarizeMessage()           # 总结单条消息
│   ├── batchSummarizeMessages()     # 批量总结消息
│   └── getSummaryStats()            # 获取总结统计
│
├── 📄 fileWatcher.js                # 文件监控服务
│   ├── watchDirectory()             # 监控目录变化
│   ├── onFileChange()               # 文件变化回调
│   └── stopWatching()               # 停止监控
│
└── 📄 timeout_checker.js            # 超时检查服务
    ├── checkTimeouts()              # 检查超时任务
    ├── registerTimeout()            # 注册超时检查
    └── clearTimeout()               # 清除超时检查
```

**重要说明**: 
- `message-reconciliation-enhanced.js` 是消息保底机制的核心
- `messageSummaryService.js` 提供消息智能总结功能
- 服务间通过事件总线 (`event_bus.js`) 通信

---

### 4. **数据库层** (`db/`)
**职责**: 数据库连接、模型定义、数据操作封装。

```
db/
└── 📄 database.js              # 数据库连接和操作
    ├── initDatabase()          # 初始化数据库 (建表、索引)
    ├── getConnection()         # 获取数据库连接
    ├── queries对象             # 所有SQL查询语句
    │   ├── messages表操作
    │   ├── bots表操作
    │   ├── groups表操作
    │   └── tasks表操作
    └── 事务管理函数
```

**数据库文件**: `backend/data/chat.db` (SQLite)
**表结构**:
- `messages` - 消息记录 (含is_pushed, is_summary字段)
- `message_summaries` - 消息总结记录
- `bots` - Bot信息缓存
- `groups` - 群组信息
- `tasks` - 任务记录

---

### 5. **WebSocket模块** (`websocket/`)
**职责**: 实时双向通信，消息推送，连接管理。

```
websocket/
└── 📄 manager.js              # WebSocket管理器
    ├── initWebSocket()        # 初始化WebSocket服务器
    ├── handleConnection()     # 处理客户端连接
    ├── sendToClient()         # 向指定客户端发送消息
    ├── broadcast()            # 广播消息到所有客户端
    └── connection池管理
```

**集成位置**: 在 `server.js` 中与HTTP服务共享同一端口。
**消息类型**: 实时消息、任务状态更新、系统通知。

## 🔧 核心服务启动流程

### 启动顺序 (在 `server.js` 中)
1. **加载配置** → 读取 `config.json` 和环境变量
2. **初始化数据库** → 调用 `db/database.js` 的 `initDatabase()`
3. **启动Express服务** → 设置中间件、路由
4. **初始化WebSocket** → 调用 `websocket/manager.js` 的 `initWebSocket()`
5. **启动业务服务** → 启动 `services/` 中的各个服务
6. **启动定时任务** → 启动 `tasks/` 中的定时任务
7. **开始监听端口** → 监听1314端口

### 服务依赖关系
```
Express服务 (HTTP API)
    ├── 依赖 → 数据库层 (db/database.js)
    ├── 依赖 → API模块 (api/openclaw.js)
    ├── 集成 → WebSocket服务 (websocket/manager.js)
    └── 调用 → 业务服务层 (services/)
```

## 🚨 常见问题与定位

### 问题1: API返回错误
**排查步骤**:
1. 查看 `server.log` 日志文件
2. 检查对应路由处理器 (在 `server.js` 中查找路由定义)
3. 检查调用的服务函数 (在 `services/` 中)
4. 检查数据库操作 (在 `db/database.js` 中)

### 问题2: 消息未实时推送
**排查步骤**:
1. 检查WebSocket连接状态 (`websocket/manager.js`)
2. 检查消息协调服务 (`services/message-reconciliation-enhanced.js`)
3. 检查数据库 `messages.is_pushed` 字段状态

### 问题3: 服务启动失败
**排查步骤**:
1. 检查端口1314是否被占用
2. 检查数据库文件 `data/chat.db` 是否存在且可读写
3. 检查 `config.json` 配置是否正确
4. 查看 `server.out` 标准输出日志

### 问题4: 性能问题
**排查步骤**:
1. 检查 `server.log` 中的响应时间
2. 检查数据库查询性能 (SQLite索引优化)
3. 检查服务层是否有阻塞操作
4. 检查内存使用情况

## 📊 监控与维护

### 日志文件
- `server.log` - 详细运行日志 (包含错误、请求记录)
- `server.out` - 标准输出日志 (启动信息、控制台输出)
- 日志位置: `backend/` 目录下

### 健康检查
- API端点: `GET /api/health`
- 返回服务状态、数据库连接状态、各服务运行状态
- 可用于监控系统集成

### 自动修复
- 通过 `auto_healer_status.json` 记录状态
- 超时检查服务自动重启失败任务
- 消息协调服务确保消息最终一致性

## 🔄 如何扩展后端功能

### 添加新API端点
1. 在 `server.js` 中添加新的路由定义
2. 实现对应的路由处理器函数
3. 如有业务逻辑，在 `services/` 中创建新服务文件
4. 如有数据操作，在 `db/database.js` 中添加SQL查询
5. 更新API文档 (`docs/` 目录下)

### 添加新业务服务
1. 在 `services/` 中创建新的服务文件
2. 导出服务函数，保持单一职责
3. 在 `server.js` 中初始化并启动服务
4. 通过事件总线与其他服务通信

### 添加新数据库表
1. 在 `db/database.js` 的 `initDatabase()` 中添加建表SQL
2. 在 `queries` 对象中添加对应的CRUD操作
3. 在相关服务中调用新的查询函数
4. 考虑数据迁移脚本 (如果需要)

## 📝 开发规范

### 代码组织
- **模块化**: 每个文件职责清晰，避免巨型文件
- **依赖注入**: 避免硬编码依赖，通过参数传递
- **错误处理**: 统一错误处理机制，不吞没异常
- **日志记录**: 关键操作记录日志，便于排查

### 文件命名
- 服务文件: `xxx-service.js` 或 `xxx-manager.js`
- 工具文件: `xxx-utils.js` 或 `xxx-helper.js`
- 测试文件: `xxx.test.js` 或 `xxx.spec.js`

### 配置管理
- 公共配置在 `config.json` 中
- 环境特定配置使用环境变量
- 敏感信息不硬编码在代码中
