## 一、现状分析 & 问题

### 1.1 当前架构

```
用户发消息 → handleSend()
     │
     ├─ POST /api/messages/{botId}  ──→  botController.sendMessage()
     │                                         │
     │                                         ├─ taskMonitor.createTask(botId, content)  ← 存入内存Map
     │                                         ├─ res.json({ taskId })  ← 立即返回前端
     │                                         │
     │                                         └─ (异步) openclawClient.sendMessage()  ← 等Bot回复
     │                                               │
     │                                               ├─ taskMonitor.updateTaskStatus('running', {currentStep: '正在发送到OpenClaw...'})
     │                                               └─ taskMonitor.updateTaskStatus('completed', {progress: 100})
     │
     └─ 前端收到 taskId → 显示"发送中"

用户点📋 → openTaskModal()
     │
     ├─ GET /api/bots/{botId}/tasks  ──→  taskMonitor.getTasksByBot(botId)  ← 从内存Map读
     └─ 每5秒轮询一次
```

### 1.2 五个 Bug

| # | Bug | 根因 | 影响 |
|---|-----|------|------|
| 1 | **已耗时永远 0ms** | 后端 `createTask` 只存 `createdAt`，不存 `startTime`；前端读 `startTime` | 用户看不到任务花了多久 |
| 2 | **进度条永远 0%** | 后端 `running` 状态时不设 `progress` 字段，只在 `completed` 时设 100% | 进度从 0% 直接跳到完成/消失 |
| 3 | **停止按钮无效** | 前端调 `POST /tasks/{taskId}/stop`，后端路由是 `POST /tasks/stop`（body传taskId），**URL 404** | 按钮点了没反应 |
| 4 | **停止了也杀不掉进程** | 后端 `stopTask` 只改内存状态为 `stopped`，不 kill 子进程，OpenClaw CLI 继续跑 | 停止是假停止 |
| 5 | **状态不实时** | `TaskMonitor` 是纯内存 Map，不 emit 事件，不推 WebSocket；只靠 5 秒轮询 | 用户看到的状态滞后 5 秒 |

## 二、产品决策（老板确认）

| 问题 | 决策 | 说明 |
|------|------|------|
| 进度条 | **去掉** | 不做模拟进度。只显示「处理中 + 已耗时」，控制权在我们手上 |
| 多任务 | **队列排队** | 底层逻辑：一条消息发给 OpenClaw 后，处理完才发下一条。弹窗展示排队列表 |
| 停止任务 | **只能停「待处理」** | 待处理的可以取消并从聊天记录中删除。正在处理中的不可停止，需提示 |
| 历史持久化 | **不需要** | 任务本质就是聊天消息，已有实时聊天记录功能存数据库 |
| 弹窗 vs 抽屉 | **保持弹窗** | 右侧已有文件树（WorkspacePanel），加抽屉会冲突。弹窗足够 |

## 四、功能修复方案

### 4.1 核心概念：消息队列

**本质认知**：每条用户消息 = 一个任务。OpenClaw 是串行处理的（一条处理完才发下一条），所以任务天然形成队列。

```
用户发 3 条消息:
  消息A → 立即发送给 OpenClaw → 状态: 处理中 (🔄)
  消息B → 等 A 完成后才发送   → 状态: 排队中 (⏳)
  消息C → 等 B 完成后才发送   → 状态: 排队中 (⏳)
```

### 4.2 后端 — TaskMonitor 重构

**改动文件**: `backend/server.js` L271-357

```
改前: class TaskMonitor { ... }                        （纯 Map，无事件，无队列）
改后: class TaskMonitor extends EventEmitter { ... }    （有事件 + 队列管理）
```

**新字段**:

```js
createTask(botId, content, model) {
  const task = {
    id: taskId,
    botId,
    content,               // 消息内容（截取前100字符展示）
    model,
    status: 'queued',      // 新状态: 'queued' | 'running' | 'completed' | 'failed'
    startTime: Date.now(),  // ← 修复 Bug1
    endTime: null,
    currentStep: '排队中',
    logs: [],              // 简约日志
    childProcess: null     // 子进程引用
  }
  this.tasks.set(taskId, task)
  this.emit('task_progress', task)  // ← 修复 Bug5: 推送给前端
  return taskId
}
```

**新方法**:

| 方法 | 作用 |
|------|------|
| `startTask(taskId)` | `queued` → `running`，记录 `startTime`，emit `task_progress` |
| `completeTask(taskId)` | `running` → `completed`，记录 `endTime`，emit `task_completed` |
| `failTask(taskId, error)` | → `failed`，emit `task_failed` |
| `cancelTask(taskId)` | 仅 `queued` 状态可调用。从 Map 中删除，emit `task_cancelled`，返回 `{ messageId }` 供删消息 |
| `setChildProcess(taskId, child)` | 存子进程引用 |
| `addLog(taskId, message)` | 追加日志（最多保留 10 条） |
| `getQueuedTasks(botId)` | 返回该 Bot 的所有 queued 任务（按创建时间排序） |

**事件监听（server.js 初始化区域新增）**:

```js
taskMonitor.on('task_progress', (task) => {
  wsManager.broadcast({ type: 'task_progress', botId: task.botId, task: sanitizeTask(task) })
})
taskMonitor.on('task_completed', (task) => {
  wsManager.broadcast({ type: 'task_completed', botId: task.botId, task: sanitizeTask(task) })
})
taskMonitor.on('task_failed', (task) => {
  wsManager.broadcast({ type: 'task_failed', botId: task.botId, task: sanitizeTask(task) })
})
taskMonitor.on('task_cancelled', (task) => {
  wsManager.broadcast({ type: 'task_cancelled', botId: task.botId, task: sanitizeTask(task) })
})

// sanitizeTask: 过滤掉 childProcess 等不可序列化字段
function sanitizeTask(task) {
  const { childProcess, ...rest } = task
  return rest
}
```

### 4.3 后端 — botController.sendMessage 改造

当前 `sendMessage` 是「创建任务 → 立刻异步调 OpenClaw」。改为支持队列逻辑：

```
改前:
  createTask → 立刻 await openclawClient.sendMessage()

改后:
  createTask(status='queued')
  → 检查该 Bot 是否有 running 任务
    → 没有: startTask → 发送给 OpenClaw
    → 有:   保持 queued，等前一个完成后自动触发
```

**核心改动**:

```js
// botController.sendMessage 异步区域改造
const taskId = this.taskMonitor.createTask(botId, content, model)

// 检查是否有正在运行的任务
const runningTasks = this.taskMonitor.getTasksByBot(botId)
  .filter(t => t.status === 'running')

if (runningTasks.length > 0) {
  // 有任务在跑，排队等待
  console.log(`⏳ [队列] Bot ${botId} 有任务在执行，当前任务排队: ${taskId}`)
} else {
  // 没有在跑的，立即执行
  this._executeTask(taskId, botId, content, model, userId, sessionId, style)
}
```

**新增 `_executeTask` 方法（私有）**:

```js
async _executeTask(taskId, botId, content, model, userId, sessionId, style) {
  this.taskMonitor.startTask(taskId)  // queued → running
  this.taskMonitor.addLog(taskId, '消息已发送到 OpenClaw')

  try {
    const result = await this.openclawClient.sendMessage(...)
    // ... 处理回复、入库、推送（现有逻辑不变）...

    this.taskMonitor.completeTask(taskId)
    this.taskMonitor.addLog(taskId, '任务完成')
  } catch (error) {
    this.taskMonitor.failTask(taskId, error.message)
  }

  // 完成后检查队列，自动触发下一个
  this._processNextTask(botId, userId, sessionId, style)
}

_processNextTask(botId, userId, sessionId, style) {
  const queued = this.taskMonitor.getQueuedTasks(botId)
  if (queued.length > 0) {
    const next = queued[0]
    console.log(`▶️ [队列] 自动触发下一个任务: ${next.id}`)
    this._executeTask(next.id, botId, next.content, next.model, userId, sessionId, style)
  }
}
```

### 4.4 后端 — 取消任务 + 删消息

**新增路由**: `DELETE /api/tasks/:taskId`

```js
// taskRoutes.js

```

**cancelTask 控制器**:

```js
cancelTask = async (req, res) => {
  const { taskId } = req.params

  const task = this.taskMonitor.getTask(taskId)
  if (!task) {
    return res.status(404).json({ success: false, error: '任务不存在' })
  }

  if (task.status === 'running') {
    return res.status(400).json({
      success: false,
      error: '该任务正在处理中，无法取消'
    })
  }

  if (task.status !== 'queued') {
    return res.status(400).json({
      success: false,
      error: '只能取消排队中的任务'
    })
  }

  // 1. 从任务队列中移除
  this.taskMonitor.cancelTask(taskId)

  // 2. 从聊天记录中删除对应的用户消息
  if (task.messageId) {
    try {
      await deleteMessageById(task.messageId)
      console.log(`🗑️ [取消任务] 已删除聊天消息: ${task.messageId}`)
    } catch (e) {
      console.warn(`⚠️ 删除消息失败: ${e.message}`)
    }
  }

  res.json({ success: true, message: '任务已取消' })
}
```


### 5.3 任务状态流转

```
  创建 → [queued] ──有空位──→ [running] ──成功──→ [completed]
              │                    │
              │ 用户取消           │ 出错
              ↓                    ↓
         [cancelled]           [failed]

规则:
  - queued → 可取消（从聊天中删消息）
  - running → 不可取消（提示「正在处理中」）
  - completed/failed → 2秒后自动从弹窗中移除
```

## 七、不做的事（明确排除）

| 不做 | 原因 |
|------|------|
| 进度条 | OpenClaw 是黑盒无法获取真实进度，模拟进度无意义 |
| 任务持久化到 SQLite | 任务 = 聊天消息，已有聊天记录系统 |
| 右侧抽屉 | 右侧已有文件树面板，会冲突 |
