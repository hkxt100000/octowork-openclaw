# 文件树的重构方案

> 目标: 参照 GenSpark 文件浏览器风格，重新设计 OctoWork 的右侧 Workspace 面板  
> 作者: AI Developer  
> 日期: 2026-04-05  
> 状态: 方案设计阶段

---

## 一、现状分析 & 问题

### 1.1 当前结构 (WorkspacePanel)

```
WorkspacePanel.vue (单聊)
├── Tab: "文件树" ← FileTreeTab.vue
│   ├── 今日文件 (数量 badge)
│   ├── 昨日文件
│   ├── 实时聊天记录
│   │   └── 历史记录 (子折叠)
│   ├── 本周文件
│   ├── 个人数字公寓 (递归树)
│   ├── 团队数字公寓 (递归树)
│   └── 公司数字公寓 (递归树)
└── Tab: "预览" ← PreviewTab.vue

GroupWorkspacePanel.vue (群聊)
├── Tab: "群聊文件树"
│   ├── 实时聊天记录
│   └── 历史记录
└── Tab: "预览"
```

### 1.2 核心问题

| 问题 | 影响 | 严重程度 |
|------|------|---------|
| **分类过多 (7个折叠区)** | 信息密度高、视觉杂乱，用户找不到目标文件 | 高 |
| **递归加载全量文件树** | `/api/bots/:botId/file-tree?maxDepth=5` 返回大量数据，首屏慢 | 高 |
| **缓存膨胀** | 为每个 Bot 缓存完整文件树到 localStorage，存储配额容易超标 | 高 |
| **Tab 只有 2 个** | "文件树"和"预览"混在一起，功能分区不清 | 中 |
| **聊天记录混在文件列表中** | 聊天记录与文件不同性质，不应混放 | 中 |
| **无面包屑导航** | 用户进入深层目录后丢失方位感 | 中 |
| **脚本文件过滤不彻底** | `filterFiles()` 白名单模式可维护性低 | 低 |

## 二、目标设计 (GenSpark 风格)

### 2.1 GenSpark 设计参考

GenSpark 文件浏览器的核心设计理念:
- **顶部 Tab 导航条**: 扁平式水平 Tab，图标 + 中文标签
- **面包屑路径**: 显示当前目录的绝对路径，支持点击跳转
- **按需加载**: 仅加载当前目录下一层，避免递归深遍历
- **简洁风格**: 白底、细边框、大间距、去卡片化

### 2.2 新导航结构设计

```
┌─────────────────────────────────────────────────────┐
│  📂 文件浏览器  │  💬 聊天记录  │  📖 教程  │      │
├─────────────────────────────────────────────────────┤
│  🏠 home > octowork > departments > ...            │
│  当前路径: /octowork/departments/tech/              │
├─────────────────────────────────────────────────────┤
│                                                     │
│  📁 agents/          →  (点击进入子目录)            │
│  📁 chat_history/    →                              │
│  📄 config.json      14:32                          │
│  📄 README.md        昨天                           │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 2.3 五个核心导航区

#### Tab 1: 文件浏览器 (默认)

- **默认根目录**: `/octowork/` (用户的工作空间根)
- **按需加载**: 点击文件夹 → 进入子目录，仅加载一层
- **面包屑**: 显示完整路径，每级可点击跳转
- **文件过滤**: 延续现有白名单过滤 (`.md`, `.txt`, `.pdf`, 图片, 视频等)，过滤 `.js`, `.ts`, `.py`, `.sh` 等脚本文件
- **操作**: 预览 / 复制路径 / 下载 / 删除
- **排序**: 文件夹在前，文件在后；按修改时间降序

**API 设计**:
```
GET /api/files/browse?path=/octowork/&depth=1
→ {
    success: true,
    currentPath: "/octowork/",
    items: [
      { name: "departments", type: "directory", modified: "...", childCount: 5 },
      { name: "config.json", type: "file", size: 1024, modified: "...", ext: ".json" }
    ]
  }
```

> **注意**: 初期可复用现有 `file-tree` API，前端做扁平化展示。

#### Tab 2: 聊天记录

- **内容**: 个人单聊 + 群聊的实时聊天记录
- **数据源**: 复用现有 `/api/bots/:botId/chat-records` 和 `/api/groups/:groupId/chat-records`
- **展示**: 按日期分组 (今日 / 昨日 / 更早)，每条显示日期、消息数、大小
- **点击行为**: 选中后自动进入预览模式
- **按 Bot/群分组**: 顶部下拉选择当前 Bot 或群组

#### Tab 3: 教程

- **内容**: 统一的用户教程/帮助文档
- **初期方案**: 一个简洁的 Markdown 文件，内嵌在前端
- **后期扩展**: 可连接后端教程 API，支持搜索和分类
- **风格**: 参照 GenSpark "教程" Tab，简洁列表 + 预览

#### 预览模式 (非独立 Tab)

- **触发条件**: 用户在任何 Tab 中点击/选中一个文件
- **行为**: 面板内容区替换为文件预览（Markdown 渲染、图片、视频、代码高亮）
- **返回**: 点击面包屑或"返回列表"按钮回到文件列表
- **与 GenSpark 一致**: 预览不是独立 Tab，是内容区的状态切换

#### 文件过滤 (延续增强)

- **白名单模式**: 保留现有过滤逻辑
- **增强点**: 将过滤配置提取到常量文件，方便统一管理
- **新增过滤**: 过滤 `node_modules/`, `.git/`, `__pycache__/` 等目录

## 六、性能优化对比

| 指标 | 当前方案 | 重构后 |
|------|---------|--------|
| 首屏 API 调用 | 4 个并发 (file-tree + yesterday + week + chat-records) | 1 个 (browse 当前目录) |
| 数据量 | file-tree 递归 5 层，可达数百 KB | 单层目录，通常 < 10 KB |
| localStorage 缓存 | 每 Bot 一个完整树，容易配额超标 | 仅缓存当前目录内容，LRU 5 条 |
| 渲染 DOM 节点 | 递归 TreeItem 组件，深层嵌套 | 平铺列表，固定一层 |
| 加载感知 | 等所有 API 完成才渲染 | 当前目录即时渲染 |

## 七、兼容性 & 迁移策略

1. **渐进式迁移**: Phase 1 完成后立即切换，旧组件保留 2 周后删除
2. **API 兼容**: 初期 `FileBrowserView` 可复用现有 `file-tree` API 数据做前端扁平化，后端新 API 就绪后切换
3. **Props 兼容**: `WorkspacePanel` 的 `botId`, `apiBase`, `fileChangeTrigger` 等 props 保持不变
4. **事件兼容**: `@select-file`, `@open-modal`, `@file-deleted` 等 emit 事件签名不变

## 八、风险 & 对策

| 风险 | 对策 |
|------|------|
| 新 Tab 结构增加学习成本 | 文件浏览器为默认 Tab，用户首次体验不变 |
| 面包屑导航深层路径过长 | 自动省略中间层级，保留首尾 |
| 后端新 browse API 开发周期 | 前端先用现有 API 数据做扁平展示，解耦开发 |
| 教程内容不足 | 初期内嵌一份简明教程 Markdown，后期逐步完善 |
