## Step 1: 新增 `_isHumanUser()` 私有方法

**目的**: 修复 Bug — 当前代码用 `mentionedId === 'user'` 判断人类用户，但系统中人类用户 ID 是 `admin`。

**位置**: `groupController.js` constructor 之后，第一个 API 方法之前（约 L40 后）

**代码逻辑**:
```javascript
_isHumanUser(id) {
  const HUMAN_USER_IDS = new Set(['admin', 'user'])
  return HUMAN_USER_IDS.has(id)
}
```

**验收标准**:
- [ ] 方法存在于 GroupController class 内
- [ ] `_isHumanUser('admin')` → `true`
- [ ] `_isHumanUser('user')` → `true`
- [ ] `_isHumanUser('dispatcher-octopus')` → `false`
- [ ] 代码语法检查通过 (`node -c groupController.js`)

---

## Step 2: 新增 `_triggerMentionedBots()` 统一触发方法

**目的**: 把"遍历 mentions → 调用 OpenClaw → 保存回复 → 广播"抽成一个共用方法，内置防循环。

**位置**: `_isHumanUser()` 方法之后

**核心逻辑**:
1. **防循环守卫**: `triggerDepth >= MAX_TRIGGER_DEPTH(1)` 时停止
2. **过滤 mentions**: 去掉发送者自己（防自环）+ 去掉人类用户
3. **构建上下文**: 读最近 10 条群聊消息
4. **逐个触发**: 调用 `openclawClient.sendMessage()` 唤醒 Bot
5. **保存回复**: Bot 回复保存到群聊 + WS 广播
6. **频率限制**: 同一对 Bot A→Bot B 5 秒冷却

**参数签名**:
```javascript
async _triggerMentionedBots({ groupId, senderId, mentions, messageContent, style, triggerDepth = 0 })
```

**验收标准**:
- [ ] 方法存在于 GroupController class 内
- [ ] 包含 `MAX_TRIGGER_DEPTH = 1` 常量
- [ ] 包含发送者自环过滤 (`id === senderId` 跳过)
- [ ] 包含人类用户过滤 (`_isHumanUser(id)` 跳过)
- [ ] 包含 OpenClaw 调用逻辑 (`this.openclawClient.sendMessage(...)`)
- [ ] 包含 Bot 回复保存 + WS 广播
- [ ] 包含频率限制 (`TRIGGER_COOLDOWN_MS = 5000`)
- [ ] 代码语法检查通过

## Step 3: 改造 `botSendToGroup` — 接入统一触发

**目的**: 核心修复 — 让 Bot @Bot 时，被 @ 的 Bot 能被 OpenClaw 唤醒。

**改动范围**: `botSendToGroup` 方法 L620-665（mentions 处理部分）

**改动内容**:
1. 人类用户通知逻辑：用 `_isHumanUser()` 替代 `mentionedId === 'user'`

3. Bot 触发逻辑：删除原来的"只发通知"代码，改为调用 `_triggerMentionedBots()`
4. 异步执行：不阻塞 API 响应

**验收标准**:

### 每个 Prompt 添加的核心内容

```
## 群聊通信协议（铁律）

⚠️ 所有对团队成员的汇报和指令，必须通过调用 send-to-group API 发送。
   只写在回复文本里的 @mention 不会触发对方，对方不会醒来。

API 调用方式：
  POST /api/bot/send-to-group
  Body: { "botId": "<自己的ID>", "groupId": <群组ID>, "content": "<消息>", "mentions": ["<被@的ID>"] }

❌ 绝对不要：只在回复文本中写 "@xxx 请审核" 就完事
✅ 必须要：调用 send-to-group API 发送消息
```
