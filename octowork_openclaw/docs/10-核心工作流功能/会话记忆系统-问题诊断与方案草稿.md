## 一、现状诊断：当前记忆管理的三大致命问题

### 问题1：清理后"保留5条"是盲选，AI直接失忆

**出处：** `octowork_session_manager.py` 第 447-449 行

```python
keep_count = CONFIG["keep_messages"]  # 5
to_archive = messages[:-keep_count]
to_keep = messages[-keep_count:]
```

**问题：**
- 直接取最后5条，不管内容是什么
- 可能全是 bot 的客套回复，没有任务上下文
- 可能是无关的闲聊，真正重要的指令在第6条被删了
- AI 重启后看到这5条，完全不知道自己刚才在干什么

**实际后果：** 用户下一句话发过去，AI 回复"您好，请问有什么可以帮您？"——失忆了。

---

### 问题2：写回的"系统摘要"等于没说

**出处：** `octowork_session_manager.py` 第 411-416 行

```python
summary_message = {
    "role": "system",
    "content": (
        f"[系统提示] 之前的对话历史已自动归档至本地记录。"
        f"以下是最近 {len(messages)} 条对话，请继续当前任务。"
    )
}
```

**问题：**
- 没说"当前任务"是什么
- 没说进度到哪了
- 没说有哪些待办项
- 没说用户最后的要求是什么
- AI 读完这句话，知道"历史被清了"，但不知道该干什么

**对比：** 人类在工作被打断后，需要的不是"你之前在工作"这句废话，而是一张便签纸上写着"正在做XX，做到第3步了，下一步是YY"。

---

### 问题3：群聊上下文完全无脑灌历史

**出处：** `groupController.js` `_triggerMentionedBots` 方法

**修复前（已改善但仍有根本问题）：**
```javascript
const recentMessages = await getGroupMessages(parseInt(groupId), 20)
// 筛选相关消息... 但这20条本身就可能全是无关的老消息
```

**问题：**
- 群聊场景下每个 Bot 被 @ 时，只拿最近 N 条消息做上下文
- 这些消息可能跨越了好几个话题
- 没有任务状态、没有该 Bot 的历史角色定位
- Bot 每次被 @ 都像"刚进群"，不知道之前讨论了什么

## 二、根因分析：为什么现在这么粗暴

```
OpenClaw 的会话机制：
  session_id → OpenClaw 服务端维护完整对话历史 → 打包发给模型API
  
我们能控制的：
  ❌ 不能控制 OpenClaw 内部的压缩策略
  ❌ 不能修改 OpenClaw 的 session 存储格式
  ✅ 能控制 "什么时候清理" 和 "清理后写回什么"
  ✅ 能控制 "发给 Bot 的上下文消息是什么"（群聊场景）
  ✅ 能在 OctoWork 侧维护自己的记忆层

核心矛盾：
  OpenClaw 管对话历史 → 我们只能在"外围"做手术
  手术做得粗 → 就是现在的盲切5条 + 废话摘要
  手术做得精 → 需要一套"记忆充电"系统（老板的新概念？）
```

## 四、待讨论：老板的"充电概念"

> （等老板输入，在这里记录讨论内容）

### 我的初步理解和猜测

"充电"可能是指：
- 清理上下文不是"断电重启"，而是"换电池" — 旧电池（长历史）拔掉，新电池（精炼摘要）装上
- 或者：AI 在空闲时主动"充电" — 把零散记忆整理成结构化的长期记忆
- 或者：每次清理前让 AI 自己做一次"记忆固化" — 用 AI 总结 AI 的对话

### 关键设计问题（等讨论）

1. **谁来做摘要？** 用另一个 AI 总结？还是被清理的 Bot 自己总结？
2. **摘要存在哪？** 写回 OpenClaw session？存在 Agent 公寓的 memory/ 目录？
3. **什么时候充电？** 清理触发时？还是定时后台跑？
4. **群聊怎么充电？** 群聊不是双人对话，多人上下文怎么为每个 Bot 单独压缩？
5. **充电内容格式？** 纯文本摘要？结构化 JSON（任务/状态/待办）？
