## 一、核心理念：让 AI 自己给自己充电

### 与大厂方案的根本区别

```
大厂做法（外科手术）：
  系统发现 token 满了 → 系统用外部 AI 摘要 → 系统替换历史 → AI 被动接受
  问题：外部AI猜"什么重要"，经常猜错

OctoWork 做法（自我充电）：
  系统发现电量低 → 系统模拟用户跟 AI 说"你要充电了" → AI 自己总结 → 系统清空并写回
  优势：没有人比 AI 自己更清楚自己在干什么
```

### 一句话方案

> **实时监控每个 Bot 的上下文电量，当电量降至 20% 以下时，充电脚本自动触发，模拟用户向该 Bot 发送"充电指令"，Bot 自行总结核心记忆，系统清空历史并写回摘要，Bot 无感继续工作。**

## 二、为什么 OctoWork 大多数 Bot 不需要担心失忆

在讲充电方案之前，先明确一个前提：**OctoWork 90% 的 Bot 失忆风险极低**，因为架构设计上已经做了防护。

### 流水线型 Bot（TokVideoGroup 等项目团队）

```
为什么不怕失忆：
├── 固定 SOP：17步工作流，每步都有明确输入输出
├── 项目守护程序：实时记录每人的工作进度，卡在哪一看便知
├── 任务卡片制度：工作以卡片形式 @指定AI，卡片本身就是上下文
├── 脚本化工具：1000+行代码的脚本，AI只需填参数，不看代码
├── A4纸原则：每个部门同时只做一个项目
└── 接力交接：做完通知下一个人，链条不断
```

这些 Bot 就像流水线工人——固定岗位、固定动作、重复执行。即使上下文清空了，拿起任务卡片就能继续干。

### 真正需要充电的 Bot（灵活沟通型）

```
需要充电的场景：
├── 与老板直聊的 Bot：话题自由、对话长、跨多个主题
├── 部门负责人：被 @到群里处理突发问题，需要理解前因后果
└── 长周期任务执行者：跨越很多轮对话的复杂任务（少见）
```

充电系统主要服务这些角色。

## 三、关键认知对齐

### 3.1 单聊和群聊对 AI 没有区别

```
用户在前端群聊里 @调度章鱼 说了一句话
  → 后端把这句话 + 上下文发给 OpenClaw
  → OpenClaw 角度：这就是一个 session 里的一条消息
  → 和单聊完全一样，只是 session_id 不同

所以：
  单聊 session_id = user_{userId}_bot_{botId}
  群聊 session_id = group_{groupId}

  充电机制对两种 session 一视同仁
  不需要区分单聊/群聊，统一按 session 监控电量
```

### 3.2 OpenClaw 有 session_status 接口可查 token 用量

```
OpenClaw 后台有 session_status 接口：
  → 直接返回当前 session 的 token 使用量（实测：精确到个位数）
  → 也可以获取 session 的消息条数、最后活跃时间等
  → 不需要我们自己估算，一个请求就拿到精确值

实测数据（2026-04-05）：
  某 Bot 使用 Google Gemini 3.1 Pro 模型
  当前 token 使用：~30,000
  模型总容量：~2,000,000
  使用率：~1.5%
  → 说明大模型的 context window 远比想象的大
```

### 3.3 电池容量取决于模型，不是固定值（v1.1 核心更新 ⭐）

```
关键发现：不同模型的 context window 差距巨大

  Google Gemini 3.1 Pro    → 2,000,000 tokens   电池巨大
  Google Gemini 2.5 Pro    → 1,048,576 tokens   电池很大
  Claude 3.5 Sonnet        →   200,000 tokens   电池大
  GPT-4o                   →   128,000 tokens   电池中等
  DeepSeek-V3 / R1         →   128,000 tokens   电池中等
  Qwen-2.5                 →   131,072 tokens   电池中等
  Llama 3 (8B)             →     8,192 tokens   电池极小

同一个 Bot 换个模型，电池容量可能差 250 倍！

电量公式必须动态感知当前模型：
  battery = (1 - current_tokens / MODEL_MAX_TOKENS[当前模型]) x 100%

  例1：Bot 用 Gemini 3.1 Pro，当前 3万 tokens
       battery = (1 - 30000 / 2000000) x 100% = 98.5%  → 🟢 满血

  例2：同一个 Bot 切到 DeepSeek，仍然 3万 tokens
       battery = (1 - 30000 / 128000) x 100% = 76.6%   → 🟡 已经黄了

  例3：某 Bot 用 Llama 3 (8K)，3万 tokens
       已经溢出 → 💀 OpenClaw 早就在后台压缩了

⚠️ 这解释了为什么有些 Bot 用 Gemini 从来不失忆，换 DeepSeek 就突然失忆
   ——不是 Bot 的问题，是电池变小了。
```

### 3.4 充电频率因模型而异

```
模型容量直接决定充电策略：

大电池模型（Gemini 2M）：
  → 日常使用几乎不可能触发紧急充电
  → 核心价值是"夜间慢充"（日常保养）和"智能减负"
  → 真正的问题不是容量不够，而是历史太长导致注意力衰减

中电池模型（GPT-4o / DeepSeek 128K）：
  → 重度使用几小时可能触发
  → 需要紧急充电 + 日常保养双重保障

小电池模型（Llama 8K 等）：
  → 几轮对话就可能触顶
  → 需要高频监控 + 激进充电策略
```

### 3.5 充电脚本 = 触发机制，不是执行者

```
充电脚本的职责：
  ✅ 定时检查所有活跃 Bot 的电量
  ✅ 发现低电量 → 触发充电流程
  ✅ 模拟用户发送充电指令
  ✅ 接收 Bot 的摘要回复
  ✅ 清空 session + 写回摘要

充电脚本不做的事：
  ❌ 不自己做摘要（让 Bot 自己做）
  ❌ 不判断什么内容重要（让 Bot 自己判断）
  ❌ 不干预正常对话流程
```

### 5.2 电量检测流程（模型感知版）

```
充电守护脚本 (battery_daemon.js)
│
├── 启动时：
│   ├── 获取所有活跃 Bot 列表
│   └── 查询每个 Bot 当前使用的模型 → 从注册表确定电池容量
│
├── 每 N 秒轮询一次（轮询频率见 5.3 节讨论）：
│   │
│   ├── 对每个活跃 session：
│   │   ├── 调用 OpenClaw session_status 接口 → 获取 current_tokens
│   │   ├── 查询该 Bot 当前模型 → 从 MODEL_MAX_TOKENS 获取 max_tokens
│   │   ├── 计算电量百分比：
│   │   │   battery = (1 - current_tokens / max_tokens) x 100%
│   │   │
│   │   │   例1：Gemini 3.1 Pro, 3万/200万 → battery = 98.5% 🟢
│   │   │   例2：DeepSeek-V3,   3万/12.8万 → battery = 76.6% 🟡
│   │   │   例3：DeepSeek-V3,  10万/12.8万 → battery = 21.9% 🔴
│   │   │
│   │   ├── 更新内存中的电量 Map
│   │   ├── 通过 WebSocket 推送电量变化给前端（含模型名称和容量）
│   │   │
│   │   └── 判断是否触发充电：
│   │       ├── battery > 60% → 🟢 安全，不处理
│   │       ├── battery 20%-60% → 🟡 预警，记录日志
│   │       └── battery < 20% → 🔴 触发充电！
│   │
│   └── 日志记录每次轮询结果
│
├── 模型切换事件监听：
│   └── Bot 切换模型时 → 立即重算电量（电池容量变了）
│       例：Bot 从 Gemini(2M) 切到 DeepSeek(128K)
│       → 之前 98% 的电量瞬间变成 76%
│       → 可能直接触发预警甚至充电
│
└── 异常处理：OpenClaw 查询失败时用上次缓存值
```

### 5.3 轮询频率设计（需探讨的性能问题）

轮询频率直接影响两个指标：**检测灵敏度** vs **系统负载**。

```
方案A：固定频率
├── 每 60 秒查一次所有活跃 session
├── 优点：实现简单
├── 缺点：Bot 数量多时负载大；低活跃 Bot 浪费查询
└── 适合：Bot 数量 < 20 的场景

方案B：分级频率（推荐）
├── 🟢 安全区 (> 60%)  → 每 5 分钟查一次
├── 🟡 预警区 (20-60%) → 每 1 分钟查一次
├── 🔴 危险区 (< 20%)  → 触发充电，不再轮询
├── 💤 闲置 (>30分钟无活动) → 暂停监控
├── 优点：资源消耗随风险级别动态调整
└── 缺点：实现稍复杂

方案C：事件驱动 + 兜底轮询（最优）
├── 每次 Bot 回复后，估算本次消耗的 token → 累加到计数器
│   （不用查 OpenClaw，几乎零成本）
├── 累加值超过阈值时 → 精确查一次 OpenClaw 确认
├── 兜底：每 10 分钟全量轮询一次，修正累加偏差
├── 优点：最省资源，响应最快
└── 缺点：估算有偏差，需要兜底校正
```

**推荐方案C**，理由：
- OctoWork 有多个 Bot 同时在线，固定频率轮询 OpenClaw 压力大
- 每次 sendMessage 后我们已经拿到了 Bot 的回复文本，可以粗算 token
- 只在接近阈值时才精确查询，其余时间零开销

### 5.4 充电触发流程

当检测到某个 session 电量 < 20%：

```
Step 1：锁定该 session（防止充电期间用户发新消息冲突）
  → 前端显示"⚡充电中..."图标
  → 用户此时发消息，排入队列等充电完成后发送

Step 2：模拟用户发送充电指令
  → 调用 openclawClient.sendMessage()
  → 消息内容（充电指令模板，见下方）
  → session_id 不变（保持上下文连贯）

Step 3：等待 Bot 回复
  → Bot 收到指令后自行总结
  → 输出结构化摘要

Step 4：提取摘要
  → 从 Bot 回复中提取核心内容
  → 保存摘要到 Agent 公寓 memory/ 目录（持久备份）

Step 5：清空 OpenClaw session
  → 调用 OpenClaw API/CLI 清空该 session 的历史

Step 6：写回摘要到新 session
  → 将摘要作为新 session 的第一条系统消息写入
  → Bot 下次对话时能看到这份摘要

Step 7：解锁 session
  → 前端恢复正常状态
  → 显示满电图标 🟢
  → 处理排队中的用户消息（如有）

Step 8：记录充电日志
  → 充电时间、Bot ID、清理前 token 数、摘要长度
  → 写入 Agent 公寓 chat_history/ 作为归档记录
```

### 5.5 充电指令模板

```
⚡ 系统通知：你的上下文记忆电量已降至 {battery}%，即将进入充电模式。

请立即执行以下操作：
1. 【当前任务】你正在处理什么任务？进度到哪了？
2. 【关键决策】对话中做过哪些重要决定？
3. 【待办事项】还有哪些事情没完成？
4. 【关键信息】有哪些数字、名称、路径等不能忘的信息？
5. 【注意事项】有什么需要特别注意的坑或风险？

请用以下格式输出（必须严格遵守）：

---充电摘要开始---
## 当前任务
（你的内容）

## 进度状态
（你的内容）

## 关键决策
（你的内容）

## 待办事项
（你的内容）

## 关键信息
（你的内容）

## 注意事项
（你的内容）
---充电摘要结束---

输出后系统将自动为你清空历史并保留这份摘要。充电完成后你将基于这份摘要继续工作。
```

### 5.6 写回摘要模板

清空 session 后，写入新 session 的第一条消息：

```
⚡ 充电完成，以下是你上次工作的核心记忆。

{Bot 自己输出的充电摘要内容}

---
充电时间：{timestamp}
充电前 Token 使用：{old_tokens} / {max_tokens}
当前 Token 使用：{new_tokens} / {max_tokens}（已释放 {released}%）

请基于以上记忆继续工作。如有遗漏可通过章鱼学院搜索历史记录。
```

## 六、前端电量显示设计

### 6.1 电量图标

```
每个 Bot 的头像/名称旁边显示一个动态电量图标：

🔋 > 60%     绿色电池图标（满格/多格）
🪫 20%-60%   黄色电池图标（少格）
🔴 < 20%     红色电池图标（闪烁）+ 文字提示"即将充电"
⚡ 充电中      闪电图标（充电动画）
```

### 6.2 数据来源（模型感知版）

```
后端通过 WebSocket 实时推送电量变化：
{
  type: 'battery_update',
  botId: 'octotech-chief',
  sessionId: 'user_admin_bot_octotech-chief',
  battery: 76.6,              // 电量百分比
  currentTokens: 30000,       // 当前使用
  maxTokens: 128000,          // 当前模型的上限
  modelId: 'deepseek-v3',     // 当前模型ID
  modelName: 'DeepSeek V3',   // 当前模型显示名
  status: 'warning'           // safe / warning / critical / charging
}

前端收到后更新对应 Bot 的电量图标。
点击电量图标可展开详情："DeepSeek V3 · 30K / 128K tokens · 76.6%"

当 Bot 切换模型时，电量会实时重算并推送更新。
```

### 6.3 仅展示，不操作

前端只做展示，不提供手动充电按钮（v1.0）。充电完全由后端自动触发和完成。用户看到的只是一个实时电量指示器。

## 十一、风险 & 应对

| 风险 | 概率 | 应对 |
|------|------|------|
| Bot 输出的摘要格式不对 | 中 | 用正则提取 `---充电摘要开始---` 到 `---充电摘要结束---` 之间的内容；失败则保存完整回复 |
| 充电过程中用户发消息 | 中 | 消息排队等充电完成（前端显示"充电中请稍候"） |
| OpenClaw token 查询 API 不可用 | 低 | 退化为估算模式（按消息长度累加） |
| 摘要本身占了太多 token | 低 | 限制摘要模板要求简洁；如超过 5000 tokens 则压缩 |
| 多个 Bot 同时触发充电 | 中 | 串行处理，排队充电，避免 OpenClaw 并发压力 |
