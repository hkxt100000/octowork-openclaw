
### 3.2 构建脚本 10 步详解

`tools/build-release.sh`（219 行）自动执行：

| 步骤 | 操作 | 说明 | 耗时 |
|------|------|------|------|
| 1 | 前置检查 | node / ncc / bytenode 是否安装 | <1s |
| 2 | 硬编码门禁 | grep 扫描源码，发现 `/Users/jason` 等立即中止 | <1s |
| 3 | 清理旧构建 | `rm -rf release/` | <1s |
| 4 | 前端构建 | `npm ci && vite build --config vite.config.web.ts` (esbuild 压缩) | ~25s |
| 5 | 后端打包 | `npx ncc build server.js -o build-tmp/ --minify` → 单文件 ~1.1MB | ~8s |
| 6 | 字节码编译 | `npx bytenode --compile build-tmp/index.js` → server.jsc ~1.9MB | <1s |
| 7 | 生成哈希 | SHA-256 防篡改校验值 | <1s |
| 8 | 生成 launcher.js | 内嵌 SHA-256 哈希的启动器（~20行） | <1s |
| 9 | 复制脚本文件 | start.sh / install.sh / VERSION / README / .env.example | <1s |
| 10 | 打包压缩 | `tar -czf octowork-chat-v{VERSION}.tar.gz` | ~2s |

**总耗时约 30-40 秒**（Mac M 系列芯片）。

```bash

#   → [2/4] 检查 ~/octowork/ 数字公寓
#   → [3/4] npm install --production (安装 bytenode + sqlite3 + dotenv)
#   → [4/4] 安装完成，打印下一步操作

# 4. 首次启动 → 获取指纹

#   → "未检测到授权文件 (license.key)"
#   → "您的机器指纹: 0e2d03f5eb955c61c4bb3a0895ccbe38"
#   → 把指纹发给管理员（微信/邮件）

# 5. 收到 license.key 后放到根目录，重新启动
cp license.key ~/octowork-chat/
./start.sh
#   → "🔑 授权校验通过 — 张三 | standard | 有效期至 2027-12-31"
#   → 浏览器打开 http://127.0.0.1:1314
```

### 5.2 用户前置条件

| 条件 | 说明 |
|------|------|

| `~/octowork/` 目录 | 数字公寓（OctoWork 核心数据），需预装 |
| `~/octowork/config/ai-directory.json` | AI 目录配置文件 |

### 5.3 版本升级（用户操作）

```bash
# 1. Ctrl+C 停止当前服务

# 2. 备份旧版本（保险起见）
mv ~/octowork-chat ~/octowork-chat-backup-$(date +%Y%m%d)

# 3. 下载并解压新版本
tar -xzf octowork-chat-v1.1.0.tar.gz -C ~/
mv ~/octowork-chat-v1.1.0 ~/octowork-chat

# 4. 恢复授权文件（重要！）

# 5. 安装依赖 + 启动
cd ~/octowork-chat
./install.sh && ./start.sh
```

**升级不影响 `~/octowork/` 里的任何数据**（聊天记录、配置、数据库全部保留）。

### 6.2 License 参数说明

| 参数 | 必填 | 说明 | 示例 |
|------|------|------|------|
| `--fingerprint` | 是 | 客户机器指纹（启动时显示的 32 位 hex） | `0e2d03f5...` |

## 七、代码保护体系

### 7.1 三层防护

```
第一层: Vite + esbuild          → 前端 JS 压缩混淆，不可读
第二层: ncc + bytenode          → 后端编译为 V8 二进制字节码，不可反编译
第三层: SHA-256 + License       → 改一个字节启动失败 + 机器指纹绑定
```

### 7.2 保护效果

| 层级 | 文件 | 用户看到的 | 能读源码？ | 能篡改？ | 能复制给别人？ |
|------|------|-----------|----------|---------|-------------|
| 前端 | `dist/*.js` | 压缩混淆 JS | 不可读 | 改了也找不到位置 | — |
| 后端 | `server.jsc` | V8 二进制 | **完全不可读** | 改 1 字节 → SHA 不匹配 → 启动失败 | — |
| 启动器 | `launcher.js` | ~20 行校验代码 | 能看但无业务逻辑 | — | — |
| 授权 | `license.key` | Base64 文本 | 能看到客户名/过期 | 改了签名对不上 | 指纹不匹配 → 启动失败 |

### 7.3 License 机制详解

**机器指纹**（6 维度哈希 — `fingerprint.js`）：
```
hostname + platform + arch + cpuModel + totalMemory + macAddress
→ SHA-256 → 取前 32 位 hex
```
同一台电脑永远相同，换台电脑完全不同。

**license.key 内容**（Base64 编码的签名 JSON）：
```json
{
  "customer": "张三",
  "company": "某某公司",
  "fingerprint": "0e2d03f5eb955c61c4bb3a0895ccbe38",
  "expire": "2027-12-31",
  "plan": "standard",
  "maxDevices": 1,
  "issuedAt": "2026-04-06",
  "signature": "hmac-sha256签名值"
}
```

**启动校验链**：
```
launcher.js
  │ 1. 读取 server.jsc，计算 SHA-256
  │ 2. 与内嵌哈希比对 → 不匹配则拒绝启动（防篡改）
  │ 3. require('bytenode') + require('./server.jsc')
  ▼
server.jsc (verifier.js 逻辑，已编译为字节码)
  │ 4. 读取 license.key → Base64 解码 → JSON 解析
  │ 5. HMAC-SHA256 验签 → 防伪造
  │ 6. 比对机器指纹 → 防复制到别的电脑
  │ 7. 检查过期时间 → 到期提醒/拒绝
  ▼
  校验通过 → 启动 Express 服务 → 托管前端 → 监听 1314 端口
```

**LICENSE_SECRET 密钥存放位置**（三处，全部受保护）：
1. `backend/src/license/constants.js` — 开发版源码（私有仓库，不交付）
2. `tools/generate-license.js` — 开发机工具（不交付）
3. `server.jsc` — 编译为字节码后（用户版，不可读取密钥）

### 7.4 License 路径兼容（已修复 — PR #22）

`constants.js` 使用双路径兜底策略，同时兼容开发和打包环境：

```javascript
// 开发环境: __dirname = backend/src/license/ → 上3级到项目根
const LICENSE_PATH_DEV = path.join(__dirname, '../../../license.key');
// 用户版(ncc打包后): __dirname = backend/ → 上1级到 octowork-chat/
const LICENSE_PATH_PROD = path.join(__dirname, '../license.key');
// 运行时自动选择存在的路径
const LICENSE_PATH = fs.existsSync(LICENSE_PATH_DEV) ? LICENSE_PATH_DEV : LICENSE_PATH_PROD;
```

## 八、自动升级检测

### 8.1 工作原理

```
octowork-chat 仓库 (GitHub/Gitee)
│
│  manifest.json              ← 版本清单（你每次发版后更新）
│
│  HTTPS GET
│  ◄─────────────────
│                     用户电脑 ~/octowork-chat/
│  返回版本信息        │  checker.js: 当前 1.0.0 < 最新 1.1.0
│  ─────────────────► │  → 通知前端显示更新提示
│                     │  → 用户手动下载新版本
```

### 8.2 后端 API

| 端点 | 说明 |
|------|------|
| `GET /api/system/version` | 返回当前版本 + Node 版本 + 平台 |
| `GET /api/system/check-update` | 拉取 manifest.json，比较版本号 |
| `GET /api/system/license-status` | 返回授权状态（客户名/过期时间，不返回密钥） |

### 8.3 检查时机

| 触发 | 方式 | 说明 |
|------|------|------|
| 后端启动 | 延迟 10s 自动检查 | 不阻塞启动流程 |
| 前端打开 | 调用 `/api/system/check-update` | 用户打开浏览器时 |
| 定时检查 | 每 6 小时 | 仅在服务运行期间 |
| 用户手动 | 设置页 "检查更新" 按钮 | 手动触发 |

### 8.4 当前升级方式（v1.0 半自动）

程序检测到新版本 → 日志/API 提示 → 用户手动下载 tar.gz → 解压替换 → 重启。

## 附录 B：素材压缩命令参考

仅在新增视频或高分辨率图片时使用，压缩后放入 `frontend/public/` 并提交到 Git：

```bash
# === 视频压缩 (ffmpeg) ===
# 目标: H.264, 720p, CRF 30-32, 音频 AAC 128k
ffmpeg -i input.mp4 -vf "scale=-2:720" \
  -c:v libx264 -crf 30 -preset slow \
  -c:a aac -b:a 128k \
  output.mp4

# === 图片压缩 (ImageMagick) ===
# 目标: 缩放到 256px 宽，质量 85%
convert input.png -resize 256x -quality 85 output.png

# === 批量压缩群组头像 ===
cd frontend/public/group-avatars
for f in *.png; do
  convert "$f" -resize 256x -quality 85 "compressed_$f"
  mv "compressed_$f" "$f"
done

# === 生成多尺寸头像 ===
convert original.png -resize 256x256 avatars/256x256/name.png
convert original.png -resize 64x64   avatars/64x64/name.png
```
