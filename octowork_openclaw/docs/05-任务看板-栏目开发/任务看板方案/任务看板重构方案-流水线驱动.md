**不需要改 schema**。现有 pipeline_state.json 已经完美包含了看板所需的全部信息：
- `pipeline[].status` → 步骤颜色/状态
- `pipeline[].executor` → 谁在做
- `pipeline[].started_at / completed_at / passed_at` → 耗时计算
- `pipeline[].reject_count / max_rejects` → 打回次数
- `event_log[]` → 时间轴/操作历史
- `guardian.last_heartbeat` → 是否卡壳

### 2.2 数据流架构

```
                         pipeline_state.json
                        (唯一真相，每个项目一份)
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                 ▼
      Guardian 心跳       Bot 完成任务      群聊 @Bot 触发
      (定时读写)          (更新 status)     (间接触发 Bot)
                               │
                               ▼
                    TaskBoxWatcher 监听文件变化
                    (chokidar watch pipeline_state.json)
                               │
                               ▼
                    WS 广播 pipeline_update
                               │
                               ▼
                    前端看板实时刷新
```

**关键设计**：前端看板**永远不写** pipeline_state.json。用户/DISP 要推进流程，通过群聊 @Bot 触发，Bot 执行后更新 pipeline_state.json → Watcher 检测到 → WS 推给前端。

看板是**只读展示层**，不是操作层。操作层是群聊。

## 三、垂直流水线 — 任务卡存放与流转

### 3.1 核心原则："空"或"1"

> **你的需求**：团队公寓和个人公寓的 in_progress 任务永远只有 **0 或 1**，保证垂直串行。

这意味着：
- 今天给视频团队 10 个视频任务，他们严格逐个完成
- 团队 task_box 里同时只有 1 个 in_progress
- 每个 Bot 个人 task_box 里同时只有 1 个 in_progress
- 完成一个 → 自动取下一个

### 3.2 文件系统布局

```
departments/TokVideoGroup/
│
├── task_box/                          ← 【团队级】：项目排队
│   ├── pending/                       ← 排队中的项目（今天10个视频，9个在这）
│   │   ├── 20260404_TokProject_02.md
│   │   ├── 20260404_TokProject_03.md
│   │   ├── 20260404_TokProject_04.md
│   │   └── ... (共 9 个)
│   ├── in_progress/                   ← 当前进行中的项目（≤ 1 个！）
│   │   └── 20260404_TokProject_01.md
│   ├── completed/                     ← 今天完成的项目
│   │   └── (空)
│   └── accepted/                      ← 已验收归档的项目
│       └── (空)
│
├── agents/04_image-production-octopus/
│   └── task_box/                      ← 【个人级】：当前工序
│       ├── pending/                   ← ASST 派发的工序卡（≤ 1 张！）
│       ├── in_progress/               ← 正在执行的工序（≤ 1 张！）
│       └── completed/                 ← 已完成的工序
│
└── project-workspace/Project/20260404/
    ├── 20260404_TokProject_01/
    │   └── 00_项目任务卡/
    │       └── pipeline_state.json    ← 这个项目的唯一真相
    ├── 20260404_TokProject_02/
    │   └── 00_项目任务卡/
    │       └── pipeline_state.json
    └── ...
```

### 3.3 "空或1"的保证机制

| 层级 | 位置 | 规则 | 谁负责执行 |
|------|------|------|-----------|
| 团队 in_progress | `departments/{dept}/task_box/in_progress/` | **最多 1 个** 项目总卡 | Octopus（ASST）调度 |
| 团队 pending | `departments/{dept}/task_box/pending/` | 可以有多个（排队中） | Octopus 创建 |
| 个人 in_progress | `agents/{botId}/task_box/in_progress/` | **最多 1 个** 工序卡 | ASST 派发时检查 |
| 个人 pending | `agents/{botId}/task_box/pending/` | **最多 1 个** 工序卡 | ASST 派发 |

**执行铁律**：ASST 在派发工序卡前，**必须检查**目标 Bot 的 `in_progress/` 是否为空。非空 = 上一个没做完 = 不许派新的。

### 3.4 完整流转过程

```
═══════════════════════════════════════════════════════════════
Step 1: Octopus（ASST）每日排程
═══════════════════════════════════════════════════════════════

  Octopus 接到用户指令："今天做 10 个视频"
  
  FOR i = 1..10:
    → 创建 project-workspace/Project/20260404/20260404_TokProject_{i}/
    → 创建 16 个子目录 (01_产品分析报告/ ... 15_成品视频/)
    → 写入 pipeline_state.json（所有步骤 blocked，第一步 ready）
    → 启动该项目的 Project Guardian（后台守护进程）
    → 在团队 task_box/pending/ 放一张项目总卡
  END

  → 取 pending/ 中排序最前的 1 个 → 移到 in_progress/
  → 群聊通知："📋 今日 10 个项目已创建，#01 开始执行"

═══════════════════════════════════════════════════════════════
Step 2: ASST 给具体 Bot 派发工序
═══════════════════════════════════════════════════════════════

  查看 pipeline_state.json → 找到 status="ready" 的下一步
  例如 intel (竞品采集) → executor = INTEL
  
  → 检查 INTEL 的 task_box/in_progress/ 是否为空
    ├─ 空 → 继续
    └─ 非空 → 等待（不派发，Guardian 会检测卡壳并汇报）
  
  → 在 INTEL 的 task_box/pending/ 放一张工序卡
  → 群聊 @intelligence-benchmark-octopus "📋 任务卡：竞品采集"

═══════════════════════════════════════════════════════════════
Step 3: INTEL 接收并执行
═══════════════════════════════════════════════════════════════

  INTEL 被 OpenClaw 唤醒 → 读取 pending/ 的工序卡
  
  → 个人 task_box: pending/ → in_progress/
  → 更新 pipeline_state.json: intel.status → "in_progress"
  → 执行实际工作（采集竞品视频、写分析报告...）
  → 输出写入 project-workspace/.../01_产品分析报告/
  → 更新 pipeline_state.json: intel.status → "completed"
  → 个人 task_box: in_progress/ → completed/
  → 群聊汇报 "@management-octopus 竞品采集完成，报告已写入"

═══════════════════════════════════════════════════════════════
Step 4: QC 质检 → ASST 推进
═══════════════════════════════════════════════════════════════

  QC 质检通过：
  → pipeline_state.json: intel.status → "passed"
  → 解锁下一步: entry_qc.status → "ready"
  → ASST 给下一个 Bot 派发工序卡（回到 Step 2）

  QC 质检打回：
  → pipeline_state.json: intel.status → "rejected"
  → ASST @INTEL 重做，附上打回原因

═══════════════════════════════════════════════════════════════
Step 5: 项目所有步骤 passed
═══════════════════════════════════════════════════════════════

  → 团队 task_box: in_progress/ → completed/
  → 从 pending/ 中取排序最前的下一个项目 → in_progress/
  → 群聊通知："🎉 #01 项目完成！#02 开始执行"
  → 开始新项目（回到 Step 2）

═══════════════════════════════════════════════════════════════

⏱ 每个时刻的保证：
  - 团队 in_progress: ≤ 1 个
  - 每个 Bot in_progress: ≤ 1 个
  - 流水线严格串行
  - 用户通过看板可以实时看到全局进度
```

## 四、Octopus 排程系统

### 4.1 Octopus 的角色

Octopus（`01_dispatcher-octopus`，即 ASST / DISP）是整个工厂的**调度中心**：

| 职责 | 触发 | 操作 |
|------|------|------|
| 每日排程 | **自动** cron + 品牌资产入库触发 | 创建当日 N 个项目文件夹 + pipeline_state.json + Guardian |
| 项目启动 | 项目创建完成时 | 将第 1 个项目移入团队 in_progress |
| 工序派发 | pipeline 中出现 `ready` 步骤 | 给对应 Bot 的 task_box 放工序卡 + 群聊 @Bot |
| 项目轮换 | 当前项目全部 passed | 将 completed 项目归档，取 pending 下一个 |
| 卡壳处理 | Guardian 报告超时 | 群聊 @相关 Bot 催促 / 上报用户 |
| 全局监控 | 随时 | 调用 `/api/board/...` 查看进度，做出调度决策 |

### 4.2 自动排程流程（已确认：全自动）

Octopus 采用**全自动排程**，不依赖用户每天手动下达指令：

```
触发源（任意一个即可）：
  A) cron 定时器 → 每天早上 9:00 自动检查品牌资产库
  B) 品牌资产入库事件 → 新产品入库时自动触发排程
  C) 用户指令 → "今天做 10 个视频"（兜底/覆盖自动判断）
                        │
                        ▼
            ┌─── Octopus 自动排程引擎 ───┐
            │                             │
            │  1. 扫描品牌资产库           │
            │     → 筛选待处理产品         │
            │     → 根据产能自动确定数量   │
            │     （或用户指令覆盖数量）    │
            │                             │
            │  2. FOR i = 1..N:           │
            │     → 创建项目目录           │
            │     → 读取部门 pipeline 模板 │
            │     → 写入 pipeline_state    │
            │     → 启动 Guardian          │
            │     → 放入团队 pending       │
            │                             │
            │  3. 取 pending #01           │
            │     → 移入 in_progress       │
            │     → 开始自动派发工序       │
            └─────────────────────────────┘
                        │
                        ▼
          团队 task_box 状态：
            pending:     N-1 个
            in_progress: 1 个 (#01)
            completed:   0 个
```

**自动产量判断逻辑**：
- 扫描 `octowork/projects/OctoWork_VisualFactory/01_品牌资产/品牌主库/` 中尚未生产过的产品
- 参考前 7 天平均完成量作为日产能上限
- 如果待处理产品 > 日产能 → 只创建日产能数量
- 用户可以随时通过群聊 @Octopus 调整（增加/减少/暂停）

## 五、Project Guardian 融合

### 5.1 Guardian 的角色

每个项目有一个 Guardian 进程（通过 OpenClaw cron/heartbeat 实现），负责：

| 职责 | 频率 | 操作 |
|------|------|------|
| 心跳 | 每 5 分钟 | 更新 `guardian.last_heartbeat` |
| 卡壳检测 | 每 5 分钟 | 检查 in_progress 步骤是否超过 `stall_timeout_minutes` |
| 自动催促 | 卡壳时 | 群聊 @相关 Bot 要求汇报进度 |
| 失忆恢复 | 卡壳 3 次 | 重新读 pipeline_state.json，向 ASST 求助 |
| 完成通知 | 全部 passed | 通知 ASST 进行项目轮换 |

### 5.2 Guardian → 看板数据流

```
Guardian 每 5 分钟更新 pipeline_state.json
                    │
                    ▼
       TaskBoxWatcher 检测到文件变化
                    │
                    ▼
        解析 JSON → 计算 progress
                    │
                    ▼
       WS 广播 { type: 'pipeline_update', ... }
                    │
                    ▼
       前端看板实时刷新 Guardian 状态
       ├── ✅ 最后心跳: 21:21:55 (3分钟前)
       ├── ⚠️ 最后心跳: 20:00:00 (26分钟前) → 显示告警
       └── 💀 超过 30 分钟无心跳 → 红色告警
```

### 5.3 看板展示 Guardian 信息

在流水线详情页的底部添加 Guardian 状态区：


### 7.3 TaskBoxWatcher 改造

**现在**：只监听 `departments/{dept}/task_box/{status}/*.md` 的增删改。

**改造后**：增加监听 `project-workspace/Project/**/**/pipeline_state.json`。

```javascript
// task_box_watcher.js 新增方法

/**
 * 监控部门的所有 pipeline_state.json
 * @param {string} deptId - 部门ID
 */
watchPipelineStates(deptId) {
    const projectsDir = path.join(this.departmentsDir, deptId, 'project-workspace', 'Project')
    
    if (!fs.existsSync(projectsDir)) return

    const watcher = chokidar.watch(
        path.join(projectsDir, '**/pipeline_state.json'),
        {
            persistent: true,
            ignoreInitial: true,
            awaitWriteFinish: { stabilityThreshold: 200, pollInterval: 100 }
        }
    )

    watcher.on('change', (filePath) => {
        console.log(`🔄 [PipelineWatcher] pipeline_state 变化: ${filePath}`)
        this.handlePipelineChange(deptId, filePath)
    })
}

/**
 * 处理 pipeline_state.json 变化
 */
async handlePipelineChange(deptId, filePath) {
    try {
        const content = JSON.parse(fs.readFileSync(filePath, 'utf8'))
        
        // 计算当前步骤
        const currentStep = content.pipeline.find(s => 
            s.status === 'in_progress' || s.status === 'ready'
        )
        
        // 计算完成进度
        const passedCount = content.pipeline.filter(s => s.status === 'passed').length
        const totalSteps = content.pipeline.length
        
        // WS 广播
        this.wsManager.broadcast({
            type: 'pipeline_update',
            deptId,
            projectId: content.project_id,
            currentStep: currentStep?.step_id,
            currentStepName: currentStep?.name,
            executor: currentStep?.executor,
            progress: { passed: passedCount, total: totalSteps },
            guardian: content.guardian,
            updatedAt: content.updated_at,
            timestamp: Date.now()
        })
    } catch (error) {
        console.error(`❌ 解析 pipeline_state 失败: ${filePath}`, error.message)
    }
}
```

**"催促"按钮逻辑**：

```javascript
// 前端点击 [📢 催促 IMGP]
async function nudgeBot(deptId, projectId, stepId, executor) {
    // 调用后端 API
    await fetch(`/api/board/${deptId}/projects/${projectId}/nudge`, {
        method: 'POST',
        body: JSON.stringify({ step_id: stepId, executor })
    })
}

// 后端处理
async nudgeBot(req, res) {
    const { step_id, executor } = req.body
    const { deptId, projectId } = req.params
    
    // 通过 botSendToGroup 在群聊中发送催促消息
    const groupId = this.getDepartmentGroupId(deptId)
    const botId = 'management-octopus'  // 以 DISP 身份发送
    const content = `📢 【看板催促】@${executor} 项目 ${projectId} 的 ${step_id} 步骤已超时，请汇报当前进度和预计完成时间。`
    
    await this.groupController.botSendToGroup({
        botId,
        groupId,
        content,
        mentions: [executor]
    })
    
    // 写入 pipeline event_log
    this.appendEventLog(deptId, projectId, {
        time: new Date().toISOString(),
        actor: 'USER',
        action: 'manual_nudge',
        step: step_id,
        detail: `用户从看板手动催促 ${executor}`
    })
    
    res.json({ success: true, message: '催促消息已发送到群聊' })
}
```

**规则**：
- 催促有频率限制：同一步骤 **5 分钟内只能催促 1 次**
- 催促消息在看板事件日志中可见（actor: USER, action: manual_nudge）
- 催促实质上是通过 `botSendToGroup` + `_triggerMentionedBots` 唤醒目标 Bot
- "跳转群聊"按钮直接 `router.push('/group/{groupId}')` 跳到对应群聊

### 8.2 看板是只读展示 + 催促快捷入口

用户在看板上**不能**拖拽/修改步骤状态。要推进流程，去群聊里操作：
- `@management-octopus Step-C 通过` → DISP 让 ASST 推进流水线

看板的数据流永远是**单向**的：pipeline_state.json → 看板显示。催促按钮走的是旁路（看板 → API → 群聊 → Bot → pipeline_state.json → 看板更新），不直接写 pipeline。

## 九、跨部门统一工厂标准

### 9.1 设计目标：绝对不硬编码部门（已确认）

TokVideoGroup 的 17 步流水线只是第一个模板。OctoRed（小红书）、OctoBrand（品牌管理）等部门有不同的流水线模板，但**框架完全一致**。

**⚠️ 核心铁律：部门和员工绝对不能硬编码！**

当前开发版 `octowork-chat-dev/` 对应的工作区有 10 个部门、62 个员工。但用户版 `octowork-chat/` 发布后，每个用户的部门数和员工数完全不同。必须做到：

- **部门列表**：动态扫描 `$OCTOWORK_WORKSPACE/departments/` 目录
- **员工列表**：动态读取 `$OCTOWORK_WORKSPACE/config/ai-directory.json`
- **流水线模板**：每个部门自带 `templates/pipeline_template.json`，没有模板的部门 = 没有流水线 = 看板显示"暂无流水线配置"
- **Bot 角色映射**：从部门的 `team_config.json` + agent 目录动态解析，不写死 INTEL/STRAT/IMGP 等角色代号

```javascript
// ❌ 错误：硬编码部门列表
const departments = ['TokVideoGroup', 'OctoRed', 'OctoBrand', ...]

// ✅ 正确：动态扫描
const workspaceRoot = process.env.OCTOWORK_WORKSPACE || path.join(os.homedir(), 'octowork')
const deptDir = path.join(workspaceRoot, 'departments')
const departments = fs.readdirSync(deptDir)
    .filter(d => fs.statSync(path.join(deptDir, d)).isDirectory())
    .filter(d => !d.startsWith('.'))  // 排除隐藏目录

// ❌ 错误：硬编码角色
const roles = { INTEL: '02_intelligence-octopus', STRAT: '03_strategy-octopus', ... }
```

### 9.2 统一 pipeline_state.json schema

所有部门共享同一个 `schema_version: "1.0"` 标准：

| 字段 | 必填 | 说明 |
|------|------|------|
| `schema_version` | ✅ | 固定 "1.0" |
| `project_id` | ✅ | `{YYYYMMDD}_{DeptPrefix}Project_{NN}` |
| `pipeline[]` | ✅ | 步骤数组（可多可少，TokVideo 17步，OctoRed 可能 7 步） |
| `pipeline[].step_id` | ✅ | 步骤唯一标识 |
| `pipeline[].name` | ✅ | 步骤名称 |
| `pipeline[].executor` | ✅ | 执行者角色代号 |
| `pipeline[].status` | ✅ | 状态（blocked/ready/in_progress/completed/passed/rejected/failed） |
| `event_log[]` | ✅ | 事件日志数组 |
| `guardian` | ✅ | Guardian 状态对象 |
| `review_policy` | 可选 | 审核策略 |
| `assets` | 可选 | 项目资产引用 |

### 9.3 部门 pipeline 模板（动态加载，不硬编码）

每个部门有自己的 pipeline 模板（放在 `departments/{dept}/templates/pipeline_template.json`）：

> **注意**：不是所有部门都有流水线。没有 `templates/pipeline_template.json` 的部门在看板上显示"暂无流水线配置"。

**TokVideoGroup 模板**（17 步）：
```
intel → entry_qc → step_a → qc_a → step_b → qc_b → step_c → qc_c → 
step_d → qc_d → step_e → qc_e → step_f → qc_f → step_g → exit_qc → publish
```

**OctoRed 模板**（7 步，示例）：
```
research → content_write → qc_content → image_create → qc_image → schedule → publish
```

**OctoBrand 模板**（5 步，示例）：
```
brand_audit → strategy_draft → qc_strategy → execution → review
```

Octopus 创建项目时，**动态读取**部门的 `templates/pipeline_template.json`，填入具体项目信息后写入 `pipeline_state.json`。如果模板不存在，Octopus 需要先提示用户或管理员创建模板。

```javascript
// Octopus 创建项目时的模板加载逻辑
const templatePath = path.join(deptDir, deptId, 'templates', 'pipeline_template.json')
if (!fs.existsSync(templatePath)) {
    throw new Error(`部门 ${deptId} 尚未配置流水线模板，请先创建 templates/pipeline_template.json`)
}
const template = JSON.parse(fs.readFileSync(templatePath, 'utf8'))
```

**每个部门独立配置 QC 角色**（已确认：每个部门都有独立 QC）：
- TokVideoGroup → `05_quality-control-octopus`（专职 QC）
- OctoRed → 可能由管理章鱼兼任 QC
- 其他部门 → 在 `pipeline_template.json` 的 `executor` 字段中指定

QC 角色不需要统一抽象层，因为 `executor` 字段已经是抽象的——流水线只关心"哪个 agent 执行这一步"，不关心它是专职 QC 还是兼任。

## 十五、7 天历史归档策略（已确认）

### 15.1 规则

> **已确认**：项目完成后保留 7 天，7 天后自动归档到 `archive/` 目录。

| 对象 | 保留期 | 归档位置 | 触发 |
|------|--------|----------|------|
| `pipeline_state.json` | 7 天 | `archive/{dept}/{YYYY-MM}/` | Octopus 每日凌晨 cron 扫描 |
| 团队 `task_box/completed/*.md` | 7 天 | 同上 | 同上 |
| 个人 `task_box/completed/*.md` | 7 天 | 同上 | 同上 |
| `project-workspace/Project/{date}/{project}/` | 7 天 | 压缩为 `.tar.gz` 存入 `archive/` | 同上 |
| 看板 API 返回的历史日期列表 | 仅展示 7 天 | 超过 7 天的日期不显示在看板 | 前端 `usePipeline.ts` 过滤 |

### 15.2 归档流程

```
每日凌晨 02:00 — Octopus 归档 cron
       │
       ▼
  扫描所有部门 project-workspace/Project/
       │
       ├── 计算项目完成时间（pipeline 最后一个步骤 passed_at）
       │
       ├── 完成时间 > 7 天前？
       │   ├── YES → 归档
       │   │   ├── 压缩项目目录 → archive/{dept}/{YYYY-MM}/{project_id}.tar.gz
       │   │   ├── 移动团队 task_box/accepted/{project}.md → archive/
       │   │   ├── 清理个人 task_box/completed/ 中相关工序卡
       │   │   ├── 写入归档日志 → archive/{dept}/archive_log.json
       │   │   └── 删除原始目录
       │   │
       │   └── NO → 保留
       │
       └── 群聊通知: "📦 已归档 N 个项目 (7天前完成)"
```
